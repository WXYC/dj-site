# Server-side origin decisions rely on Cloudflare pinning the request Host

Several server routes read the incoming request's origin — `request.url`, `request.nextUrl`, and the `Host` / `x-forwarded-host` headers — when they build a redirect target, scope a cookie, or check that a request is same-origin. `request.url` reflects the client-sent `Host` header, which is not cryptographically bound to anything: a client that reaches the server directly can put any value there. This ADR records why we deliberately trust that value on this deployment, and where that trust does and does not matter. It resolves [#844](https://github.com/WXYC/dj-site/issues/844) (surfaced by the [#839](https://github.com/WXYC/dj-site/issues/839) review of [#597](https://github.com/WXYC/dj-site/issues/597)).

## The trust boundary

dj-site ships as a single Cloudflare Worker (OpenNext, Direct Upload via `wrangler` — see [`wrangler.jsonc`](../../wrangler.jsonc) and [`docs/ci-cd.md`](../ci-cd.md)). Cloudflare invokes the Worker **only** for hostnames routed to it: the production custom domain (`dj.wxyc.org`), per-commit preview subdomains (`*.wxyc-dj.pages.dev`), and the `*.workers.dev` fallback. All of those are hostnames the WXYC Cloudflare account owns. A request carrying an arbitrary `Host` (`evil.example`) is not routed to this Worker at all — there is no path by which it executes with an attacker-chosen origin. In effect **Cloudflare pins the request `Host` to the set of account-owned origins.** No separate reverse proxy sits in front; the edge itself is the pin.

There is intentionally **no env-pinned canonical frontend origin** to derive redirects from instead. The only origin env vars are backend/auth targets (`NEXT_PUBLIC_BACKEND_URL`, `NEXT_PUBLIC_BETTER_AUTH_URL`, server-only `AUTH_REWRITE_URL`; see [`docs/env-vars.md`](../env-vars.md)). Redirects are built request-relative on purpose: the same code must land the browser back on whichever owned origin actually served the request — production *and* the preview deploys that share the codebase — so hardcoding or env-pinning a single origin would be wrong for the preview model, not safer.

## Why the auth routes are safe under this boundary

- **`app/auth/verify-email/route.ts`** resolves `callbackURL` against `request.url` and 302s the browser there. `safeCallbackPath()` first constrains `callbackURL` to a root-relative path (rejecting absolute, `//host`, and `/\host`), then re-checks the resolved origin against `request.url`'s origin. That re-check is a **self-consistency** comparison — both sides derive from the same `request.url`, so a spoofed `Host` is spoofed identically on both and the "reject off-origin" property still holds; its job is to catch encoded-slash tricks the URL parser would normalise into `//host` (#597), not to assert a trusted absolute origin. The final redirect's *host* does come from `request.url`, but (a) the value is already root-relative, so `Host` can only change which owned origin the browser returns to, never redirect off-app, and (b) a `Host` is only spoofable on the attacker's *own* request — an attacker cannot make a victim's browser send a forged `Host` to `dj.wxyc.org`, and their own request carries no victim session or verification token. Not victim-exploitable; see #844.
- **`app/auth/[...path]/route.ts`** reads only `new URL(request.url).search` and forwards to an **env-pinned** upstream base. The request origin never influences the upstream target.
- **`lib/features/session-guards.ts`** (`guardAppStateMutation`) compares the request's `Origin`/`Referer` host against its own served host and additionally requires a session. That is a relative comparison that stays self-consistent under a spoofed `Host`, and the cross-site CSRF case it defends against cannot forge the victim browser's `Host` anyway.

## Consequences

- If dj-site is ever fronted by an additional proxy, or moved off Cloudflare's route-by-`Host` model, this assumption must be re-validated: security-relevant origin decisions would then need a validated `Host` allowlist or a pinned canonical origin. Until then, no code change is warranted and none was made for #844 — pinning would break the request-relative preview model without closing an exploitable gap.
- New server routes that build an absolute redirect/URL from `request.url` or set a cookie scoped by request origin inherit this boundary. Keep them either request-relative-and-root-relative-validated (like verify-email) or env-pinned (like the auth proxy); do not treat `request.url`'s host as an independently trusted absolute origin.
